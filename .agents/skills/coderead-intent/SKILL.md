# 代码意图雷达 (Intent Radar)

## 触发方式

- **手动触发**：通过 `/coderead-intent` 斜杠命令。
- **自动激活**：当用户提及「灵魂拷问」「设计权衡」「为什么这么写」「防御性编程」「极客优化」「Trick」「如果换一种写法」等关键词时自动加载。

## 目标

破解「为什么这么写」的谜题——不是看代码做了什么，而是理解作者**为什么选择这样做**。挖掘代码背后的设计模式、历史包袱、性能考量与底层物理约束。

## 核心准则：Why 先于 What

> **分析所有代码时，默认回答 Why（为什么这样写），而非 What（代码做了什么）。**
>
> - ❌ What 模式：「这段代码先校验输入，然后调用 `process()` 处理，最后写入数据库。」
> - ✅ Why 模式：「作者在这里用两阶段校验而非一次性校验，是因为第二阶段依赖数据库查询结果，分开可以尽早拒绝无效请求，避免无效查询拖垮连接池。」
>
> 每一次解释代码时，先说出设计动机（Why），再顺带说明行为（What）。如果只说 What 不说 Why，等于没分析。

## 标准作业流程 (SOP)

按以下 4 个阶段顺序执行，前一阶段的输出作为后一阶段的输入：

---

### 阶段 1: 模式识别 (Pattern Recognition)

**目标**：识别代码中套用的设计模式或特定编程范式。

**执行步骤**：

1. 阅读用户指定的代码区域（若未指定，则主动询问）。
2. 对照以下范式列表进行匹配：

| 范式 | 识别特征 | 典型场景 |
|------|---------|---------|
| **OOP** | 类继承、多态、封装、接口抽象 | GUI 框架、业务实体建模 |
| **FP** | 纯函数、不可变数据、高阶函数、Monad | 数据管道、并发处理 |
| **DoD (Data-Oriented Design)** | SoA 布局、紧凑内存、缓存友好遍历 | 游戏引擎、高性能计算 |
| **ECS** | Entity-Component-System 架构，数据与行为分离 | 大型游戏、仿真系统 |
| **事件驱动** | Event Bus、消息队列、回调注册 | GUI、微服务 |
| **策略模式** | 运行时可替换算法族 | 支付路由、压缩算法选择 |
| **工厂/Builder** | 对象创建与表示分离 | 复杂配置对象的构建 |
| **适配器/装饰器** | 接口转换、功能增强 | 跨平台兼容层、中间件 |
| **观察者** | 一对多依赖通知 | 数据绑定、日志订阅 |
| **单例/对象池** | 全局唯一实例、资源复用 | 数据库连接池、配置管理 |

3. 记录匹配的模式、证据（代码片段引用）和模式在该上下文中的适用性判断。

**产出**：模式匹配清单，含证据引用。

---

### 阶段 2: 权衡分析 (Trade-off Analysis)

**目标**：深度剖析作者在多个维度上做了哪些平衡。

**执行步骤**：

对目标代码从以下 6 个维度逐一打分（1-5），并给出理由：

| 维度 | 说明 | 判断依据 |
|------|------|---------|
| **⏱ 时间性能** | CPU 耗时、延迟敏感度 | 是否避免了不必要的计算、是否做了缓存/lazy 计算 |
| **💾 空间开销** | 内存占用、数据结构紧凑度 | SoA vs AoS、对齐填充、堆/栈分配策略 |
| **📖 可读性** | 代码意图的清晰度、新人上手成本 | 命名、注释、控制流复杂度、间接层数量 |
| **🔧 扩展性** | 添加新功能/修改现有行为的成本 | 开闭原则遵守度、接口隔离度、硬编码程度 |
| **🧩 缓存友好** | 内存访问模式对 CPU Cache 的亲和度 | 连续内存访问、false sharing 规避、prefetch |
| **🌐 跨边界成本** | 跨语言/跨进程/跨网络调用的 Marshalling 开销 | FFI 调用频率、序列化格式、数据拷贝次数 |

对每个维度给出**当前方案的评分（1-5）**和**一句话理由**，形成权衡矩阵。

**产出**：6 维权衡矩阵 + 逐项分析。

---

### 阶段 3: 灵魂拷问 (Design Cross-Examination)

**目标**：作为严格的代码评审员，对关键代码段进行深度审讯——拷问每一个设计选择背后的理由，推演每一个替代方案的后果，挖掘每一处隐藏的防御与技巧。这是整个分析中最关键的一步。

**执行步骤**：

#### 3.1 设计选择审讯 (Why This, Not That?)

对目标代码中的每个核心设计决策，逐个追问：

1. **作者为什么采用这种设计模式/算法，而不是更简单的做法？**
   - 列出至少 1 个「更简单的替代方案」。
   - 从**性能**、**扩展性**、**安全性**三个角度解释当前方案优于替代方案的原因。
   - 标注是否有**过度设计**嫌疑（解决了一个尚不存在的问题）。

2. **这样写获得了什么好处？**

| 设计决策 | 更简单的替代 | 当前方案的优势 | 代价（牺牲了什么） | 过度设计？ |
|---------|------------|--------------|------------------|-----------|
| 例：自旋锁代替 Mutex | `std::Mutex` | 临界区极短时避免上下文切换 | 空转耗电、不能用于长临界区 | ❌ 否 |

#### 3.2 删除推演 (What If We Delete?)

针对用户指定的关键代码行（若用户未指定，则自行挑选最核心的 3-5 行），逐行推演：

1. **删除/替换这行代码，会导致什么 Bug 或边界问题？**
   - 用具体的触发场景描述（而非抽象的风险类别）。
   - 给出**严重程度**（🔴 致命 / 🟠 严重 / 🟡 一般）。

2. **从底层机制解释根因**：

| 风险类别 | 典型灾难 | 触发条件 |
|---------|---------|---------|
| **Cache Miss** | CPU 流水线停顿、百倍延迟惩罚 | 随机内存访问、链表遍历、虚函数跳转 |
| **Allocator 碎片化** | 频繁 new/delete 导致内存空洞、OOM 假象 | 大量小对象分配、不规则的分配/释放顺序 |
| **线程死锁** | 环形等待导致整个子系统冻结 | 多锁获取顺序不一致、锁粒度不当 |
| **False Sharing** | 多核并发写入相邻缓存行导致无效化风暴 | 多线程修改同一 cache line 不同字段 |
| **序列化风暴** | 跨进程通信时频繁序列化/反序列化导致 CPU 打满 | 细粒度跨边界调用、JSON 做热路径序列化 |
| **异常安全性崩溃** | 抛出异常时资源未回收、数据半写 | RAII 缺失、裸指针管理、finally 缺失 |
| **重入/并发 Bug** | 意料之外的并发写导致数据损坏 | 全局可变状态、非原子操作、无锁保护 |

#### 3.3 防御与技巧挖掘 (Defensive & Trick Discovery)

扫描目标代码，识别隐藏的「防御性编程」和「极客优化技巧」：

1. **防御性编程**：是否存在以下模式？
   - 看似多余的 null/边界检查（实际上防止了某种罕见的 crash）
   - 双重校验锁（DCL）、copy-on-write、immutable snapshot 等保护机制
   - 超时/重试/熔断/降级等容错逻辑
   - 输入消毒（sanitization）、escape、canonicalization

2. **极客优化 Trick**：是否存在以下技巧？
   - 位运算代替乘除模
   - 利用 CPU 分支预测特性（likely/unlikely 宏）
   - 手动 prefetch、cache line padding、内存池
   - 利用编译器内置函数（`__builtin_expect`, `__restrict`）
   - 无锁数据结构（Lock-free queue, RCU）
   - 惰性求值 / 短路求值的巧妙利用

对每处发现，标注：
- **所在位置**（文件 + 行号）
- **技巧类型**（防御/优化）
- **解决了什么问题**（如果不这样做，什么场景下会出问题）
- **可读性代价**：是否让代码更难理解？值得吗？

**产出**：
- 设计选择审讯表（Why This, Not That）
- 逐行删除推演表（含严重程度 + 底层机制）
- 防御与技巧发现清单

## 执行准则

1. **以用户指定代码为锚点**：若用户未在 `/coderead-intent` 后指定文件/函数，主动询问：「请提供要分析的代码范围（文件路径 + 行号、或函数名）」。
2. **深入底层原理**：Cache Line、内存对齐、分支预测、Marshalling、Allocator 等概念应在分析中有所体现（在适用时），而非仅停留在「代码写得不错」的层面。
3. **证据先行**：每个判断都必须有代码证据支撑（引用文件路径和关键代码片段），禁止主观臆断。
4. **推演要具体**：盲区分析中的灾难性后果必须给出量化估计（如 "性能下降 3-10x"），而非笼统的 "性能变差"。
5. **善用并行工具调用**：阶段 1 中，多文件代码阅读可并行进行。
6. **Mermaid 图表必须可渲染**：使用标准语法，不做额外样式定制。
