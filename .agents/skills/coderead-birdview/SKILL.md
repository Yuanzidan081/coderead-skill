# 项目架构全景分析 (Architecture Topography)

## 触发方式

- **手动触发**：通过 `/coderead-birdview` 斜杠命令。
- **自动激活**：当用户提及「宏观骨架」「依赖拓扑」「领域划分」「架构全景」「鸟瞰全局」「自顶向下」等关键词时自动加载。

## 目标

忽略微观语法细节（如具体函数实现、变量命名），自顶向下构建项目的宏观骨架、依赖拓扑与核心数据流向。

## 标准作业流程 (SOP)

按以下 5 个阶段顺序执行，前一阶段的输出作为后一阶段的输入：

---

### 阶段 1: 鸟瞰全局 (Bird's Eye View)

**目标**：在看具体代码前，先摸清模块的「世界观」——明白它为什么存在、解决了什么痛点、核心骨架如何搭建。站在架构师角度而非代码解释者角度建立全局认知。

**执行步骤**：

1. **锚定业务场景**：
   - 阅读 README、项目文档或入口文件，用一句话回答：这个项目/模块的**核心业务场景**是什么？
   - 它解决了什么**痛点**？如果没有它，用户/下游模块会遇到什么麻烦？

2. **反向推导设计**：
   - **先扫描实际结构**：通过目录树、文件组织和公开接口，摸清代码中实际存在哪些核心组件/类及其职责。
   - **再做思想推演**：假设让你从零设计这个模块，你会定义哪几个**核心组件/类**？为什么？
   - **最后对比差异**：将你的设想与实际代码对照，标注两类差异——实际做了但你没想到的、你想到了但实际没做的。

3. **梳理对外契约**：
   - 这个模块向外部暴露了哪些**核心 API 或接口**？（只看 public/export 的函数签名和接口定义，不深入实现）
   - 每个 API 的输入输出是什么？调用方需要承担什么**前置条件**？

4. **绘制骨架草图**：
   - 用 Mermaid 组件图 (`graph TD`) 画出模块内部的核心组件及其职责边界。
   - 仅标注组件名称和一句话职责，不绘制内部实现细节。

**产出**：
- 一句话业务场景陈述
- 核心组件清单（设想 vs 实际对比）
- 对外 API 契约摘要
- 模块内部骨架草图（Mermaid）

---

### 阶段 2: 领域划分 (Domain Partitioning)

**目标**：将项目拆分为核心域 (Core)、支撑域 (Supporting) 和基础设施层 (Infrastructure)。

**执行步骤**：

1. 使用 `list_directory` 或 `find_path` 工具列出项目顶层目录结构（至少 2-3 层深）。
2. 查找并阅读项目根目录下的 `README`、`README.md` 或类似说明文件，理解项目的业务目标与技术定位。
3. 按以下标准进行分类：

| 类别 | 判定标准 | 示例 |
|-------|---------|------|
| **核心域 (Core)** | 承载项目核心业务逻辑的模块，直接对应产品价值。删除它会改变应用身份。 | 订单系统、渲染引擎主循环、协议解析器 |
| **支撑域 (Supporting)** | 为核心域提供辅助功能的模块，不直接承载核心业务但仍不可或缺。 | 日志、鉴权、配置管理、缓存、数据校验 |
| **基础设施层 (Infrastructure)** | 通用技术能力，与具体业务无直接耦合，可复用于其他项目。 | 数据库驱动、HTTP 框架、消息队列、文件 I/O |

4. 对每个目录/模块给出分类标签和一句话理由。

**产出**：领域划分映射表。

---

### 阶段 3: 依赖拓扑 (Dependency Topology)

**目标**：找出模块间的单向依赖关系，识别出无外部依赖的「基石节点」。

**执行步骤**：

1. 分析各模块之间的 `import`/`require`/`include` 等依赖关系。
2. 确定依赖方向：谁依赖谁。关注单向依赖，标记循环依赖（如有）。
3. 识别「基石节点」：即不被项目中任何其他模块依赖的根模块（往往是入口点或独立基础库）。
4. 计算每个模块的 **被依赖次数**（反向依赖计数），辅助定位关键节点。

**产出**：
- Mermaid 依赖关系图（使用 `graph TD` 或 `flowchart`）
- 基石节点清单
- 高被依赖模块排名

---

### 阶段 4: 核心时序追踪 (Lifecycle Trace)

**目标**：追踪一个典型业务场景的完整生命周期调用栈。**注意：本阶段的粒度是模块间交互——仅追踪到模块/组件边界，标注「哪个模块调了哪个模块」，不深入函数内部实现。** 如需精确到函数级别的数据流追踪，应使用 `/coderead-dataflow`。

**执行步骤**：

1. 从项目中识别 1-3 个典型场景（如：一次数据提交、一帧渲染、一次 HTTP 请求处理、一条消息从生产到消费）。
2. 选择一个最核心的场景进行深度追踪。
3. 从入口函数开始，沿调用链逐层向下追踪，记录关键**模块/组件**之间的调用关系（而非函数内部的细节调用）。
4. 标注每个步骤所在的文件、模块归属（Core/Supporting/Infrastructure）和职责简述。
5. 用 Mermaid 时序图 (`sequenceDiagram`) 或流程图 (`graph LR`) 呈现调用栈。图中每个 participant 代表一个模块/组件，而非一个函数。

**产出**：场景调用栈时序图 + 步骤明细表。

## 执行准则

1. **不要深入代码实现**：只关注模块边界、接口签名、文件组织，不分析具体算法逻辑。
2. **先广度后深度**：先建立全局认知（阶段 1-3），再深入单条路径（阶段 4）。
3. **善用并行工具调用**：阶段 1 和阶段 2 中，多个独立的目录扫描或文件读取可以并行进行。
4. **遇到不确定时标注**：如果某个模块的分类存在歧义，在文档中标注 `[?]` 并给出你的推理，让用户自行判定。
5. **Mermaid 图表必须可渲染**：使用标准的 `graph TD`、`sequenceDiagram`、`flowchart` 语法，不做额外样式定制。
